# CI/CD Review Findings

**Reviewed:** 2026-03-15
**Reviewer:** Automated analysis (read-only)
**Scope:** `.github/workflows/`, `Makefile`, `.pre-commit-config.yaml`, `Dockerfile*`, `docker-compose.yml`, `requirements.txt`, `requirements.lock`, `.gitignore`, `CONTRIBUTING.md`

---

## Critical (blocking clean v0.8.0 public release)

**1. GitHub Actions smoke workflow does not run the full CI gate — and CONTRIBUTING.md documents this incorrectly.**

The smoke workflow (`.github/workflows/smoke.yml`) runs only `pytest tests/test_smoke.py -v`. The actual CI gate (`make ci`) runs smoke + testlab suite + UI tests (Playwright) + bandit + pip-audit + URI secret scan. CONTRIBUTING.md states: "`.github/workflows/smoke.yml` runs `pytest tests/test_smoke.py` on every push and every PR targeting `main`." This is accurate but incomplete: it implies smoke-only is the CI gate, when in fact `make ci` is the required gate. A passing smoke.yml check gives false confidence; a PR could break the testlab or UI suite and still show green in GitHub.

Impact: Contributors see a green Actions badge on a PR that would fail `make ci`. The cross-suite event loop pollution footgun (documented in CLAUDE.md) is precisely the failure mode that only appears when all three suites run together. It cannot be caught by smoke-only CI.

Recommended fix: Replace or supplement the smoke workflow with a full `make ci`-equivalent workflow, or add a second workflow that runs testlab + UI on PRs to `main`. Alternatively, split into two jobs — smoke (fast, all pushes) and full-ci (PRs to `main` only).

---

**2. PRs to `dev` have no GitHub Actions gate whatsoever.**

`smoke.yml` triggers on `pull_request: branches: [main]`. The branch strategy routes all feature PRs through `dev` first (feature → dev → main). A broken feature branch merged into `dev` gets no automated check before it accumulates with other changes and is presented as a single PR to `main`. The smoke workflow fires on all pushes (including to feature branches) but not on PRs to `dev`.

Impact: The integration branch (`dev`) is unprotected by CI. Bugs compounding across feature merges to `dev` are only caught when the `dev → main` PR runs smoke — which still misses testlab and UI.

Recommended fix: Add `dev` to the `pull_request: branches:` trigger in smoke.yml, or add a separate `ci-pr-to-dev.yml` workflow.

---

**3. CI installs from `requirements.txt` (unpinned ranges), not `requirements.lock` (pinned).**

`smoke.yml` line 31: `pip install -r requirements.txt`. The lock file (`requirements.lock`, 143 lines of pinned transitive deps) exists and is generated by `make lock`, but the CI workflow ignores it. Any upstream package publishing a breaking minor release within a semver range (e.g., `langchain~=1.0` or `fastapi~=0.132`) can silently break CI at the next workflow run with no code change.

Impact: Non-reproducible CI builds. A passing test run today may fail tomorrow due to upstream dependency churn, unrelated to any code change.

Recommended fix: Change CI `pip install` to use `requirements.lock` (`pip install -r requirements.lock`). For the public release, reproducibility is a baseline requirement.

---

**4. `make briefing` hardcodes `jp-cruz/LegionForge` — must be updated before v0.8.0 repo transfer.**

`Makefile` line 177: `gh pr list --state open -R jp-cruz/LegionForge`. After the v0.8.0 repo transfer to `LegionForge/LegionForge`, this will silently fail (`gh: repository not found`) and show `(gh not configured or no open PRs)`. The error is suppressed with `2>/dev/null`.

Impact: `make briefing` becomes silently wrong after transfer. Not a CI failure, but a developer experience regression that will cause confusion.

Recommended fix: Replace the hardcoded repo reference with `gh pr list --state open` (auto-detect from `git remote`) or parameterize: `GH_REPO ?= $(shell gh repo view --json nameWithOwner -q .nameWithOwner 2>/dev/null)`.

## High (important for maintainability)

**5. No automated smoke test count floor enforcement in CI.**

CONTRIBUTING.md documents a hard floor of 2247 smoke tests ("never goes down"). CLAUDE.md reinforces this. There is no automated check in `smoke.yml` or any Makefile target that asserts `pytest --collect-only | wc -l >= 2247`. If a refactor accidentally deletes or deselects tests, the floor violation will not be caught by CI — the suite will simply report fewer passing tests, and the run will still exit 0.

Impact: The documented invariant ("smoke test count is a floor") is not enforced by tooling. It is enforced only by convention and code review.

Recommended fix: Add a step to the smoke CI workflow that extracts the collected test count and fails if it falls below the documented baseline. A one-liner: `pytest tests/test_smoke.py --collect-only -q 2>/dev/null | tail -1 | awk '{if($1 < 2247) exit 1}'`.

---

**6. No CI workflow for Docker image builds — images are never validated in CI.**

Six Dockerfiles exist (`Dockerfile.gateway`, `Dockerfile.testlab`, `Dockerfile.sandbox`, `Dockerfile.agent-base`, `Dockerfile.analyzer`, `Dockerfile.pentest`, plus `guardian/Dockerfile`). None are built in GitHub Actions. A broken `COPY` path, a missing dependency in `requirements.txt`, or an incompatible base image update will only be caught when a developer manually runs `make docker-build` locally.

The gateway Dockerfile (`Dockerfile.gateway`) notably has no `HEALTHCHECK` directive, unlike the guardian Dockerfile which has one. The docker-compose.yml guardian service has a `healthcheck` block, but the gateway container does not.

Impact: Docker build failures are invisible to CI. A broken gateway image will not be caught until deployment.

Recommended fix: Add a `docker-build.yml` workflow that builds all images (no push, cache-from registry) on PRs to `main`. Separately, add `HEALTHCHECK` to `Dockerfile.gateway`.

---

**7. No CI check for Black formatting (`make lint`) — pre-commit hook is the only gate.**

The pre-commit config includes Black (`rev: 24.3.0`). But pre-commit hooks are opt-in per developer and must be installed (`pre-commit install`). Nothing in CI verifies that committed code is Black-formatted. A developer who skips pre-commit installation (or force-commits with `--no-verify`) can merge unformatted code without CI catching it.

Impact: Formatting violations accumulate silently until someone runs `make lint` locally. The Black version in `.pre-commit-config.yaml` (`24.3.0`) also diverges from `requirements.txt` (`black>=26.3.1` — bumped for CVE-2026-32274), creating a version split between pre-commit and the project formatter.

Recommended fix: Add `black --check src/ tests/ config/` as a CI step in the smoke workflow, using the version from the venv (not a separate pre-commit install). Align the pre-commit `rev` with the `requirements.txt` constraint.

---

**8. `bandit` configuration is entirely absent — no `.bandit` config file, no `[tool.bandit]` section.**

`make security-audit` runs `bandit -r src/ -ll` (medium+ severity). There is no `bandit.yaml`, `.bandit`, or `[tool.bandit]` section anywhere. This means:
- Suppressions are handled inline with `# nosec B608` comments (15 occurrences in source) but there is no central allowlist of expected findings.
- The severity/confidence threshold (`-ll`) is baked into the Makefile string, not version-controlled as a config file.
- Adding bandit to a CI workflow without a config file risks false-positive failures breaking CI after a bandit version update that adds new checks.

Recommended fix: Add a `pyproject.toml` (or `bandit.yaml`) at the project root with `[tool.bandit]` configuration declaring skips, targets, and severity level. This also enables IDE integration.

---

**9. `review-prep` has a step numbering bug.**

`Makefile` line 1137: `─── [4/6] Dependency CVE scan ───` is labeled `[4/6]` in a 7-step sequence where all other steps are labeled `[N/7]`. The sequence runs `[1/7]`, `[2/7]`, `[3/7]`, `[4/6]`, `[5/7]`, `[6/7]`, `[7/7]`. This is a cosmetic defect but signals incomplete review of the `review-prep` target.

---

**10. No automated CHANGELOG update — the `[Unreleased]` section is permanently stale.**

`CHANGELOG.md` uses Keep a Changelog format with an `[Unreleased]` section. There is no Makefile target, pre-commit hook, or GitHub Action that validates whether the `[Unreleased]` section has been updated when code changes land. The section has been empty (`## [Unreleased]\n\n---`) since at least the current review date.

Impact: The changelog becomes a manual artifact that developers forget to update. For a public release at v0.8.0, a complete changelog is expected by open-source consumers.

Recommended fix: At minimum, add a CI step that fails if `[Unreleased]` section is empty when the PR modifies `src/`. A `make changelog-check` target wrapping a `grep` check is sufficient. For automation, consider `git-cliff` or `commitizen` (both support conventional commits).

## Medium (quality of life)

**11. `make ci` and `make security-audit` both run the full test suite — `make ci` runs it twice.**

`make ci` calls `make test` then `make security-audit`. `make security-audit` also calls `make test` internally (line 1088). The full three-suite test run (~70s) executes twice per `make ci` invocation. This is invisible to developers unless they read the Makefile carefully.

Recommended fix: Refactor `security-audit` to accept a flag that skips the test run when called from `make ci`, or restructure the dependency so `ci` calls test once and then calls only the non-test portions of the security audit (bandit + dep-audit + URI scan).

---

**12. No CI hardware profile for `ubuntu-latest` — the `mac_m4_mini_16gb` profile is used in CI.**

`smoke.yml` sets `AGENT_HARDWARE_PROFILE: mac_m4_mini_16gb`. The hardware profile (`config/hardware_profiles/mac_m4_mini_16gb.yaml`) contains macOS-specific settings: `keychain_access_allowed: true`, Ollama model paths under `/Volumes/MAC_MINI_1TB/`, Metal GPU settings. Most of these are no-ops in smoke tests (no live Ollama, no DB), but loading a macOS-specific profile in a Linux CI environment is fragile. A test that reads profile fields for assertion will produce macOS-specific values on Linux CI.

A `ci_ubuntu.yaml` hardware profile already has a precedent in the design — `keychain_access_allowed: false` is called out as the CI setting in the profile comments (line 186).

Recommended fix: Add a `ci_ubuntu.yaml` hardware profile with `keychain_access_allowed: false`, no macOS paths, and use it in GitHub Actions: `AGENT_HARDWARE_PROFILE: ci_ubuntu`.

---

**13. Docker builds have no layer cache strategy in Makefile targets.**

`make build-gateway` (line 968): `docker build -f Dockerfile.gateway -t legionforge-gateway:latest .`
No `--cache-from` flag, no BuildKit cache mounts. Each local build reinstalls all Python dependencies from scratch unless Docker's layer cache happens to hit. In CI this means a cold install on every workflow run.

Recommended fix: Enable BuildKit (`DOCKER_BUILDKIT=1`) and add `--cache-from` against a registry tag in CI workflows. For local builds, at minimum add `DOCKER_BUILDKIT=1` to the Makefile docker build commands to enable BuildKit's improved layer caching.

---

**14. `.env.example` is actually a TaskMaster-generated file, not a LegionForge-native one.**

`.env.example` (12 lines) lists `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GITHUB_API_KEY` etc. with format strings (`sk-ant-api03-...`, `ghp_...`) — these are TaskMaster AI patterns, not LegionForge's secret management model. LegionForge explicitly uses macOS Keychain for all secrets; `.env` only holds non-sensitive config (`AGENT_HARDWARE_PROFILE`, `LOG_LEVEL`, etc.). The `.env.example` is misleading and contradicts the project's security posture for new contributors.

`.env.secrets.example` (10 lines) is the correct template, but it is less discoverable.

Recommended fix: Rewrite `.env.example` to document the actual non-sensitive config variables (matching the real `.env` contents), with a clear note that secrets go through Keychain. Optionally, remove or rename `.env.example` to avoid confusion.

---

**15. `make publish-preview` has no CI counterpart and is a manual-only step.**

`docs/public/` is gitignored (line 107 of `.gitignore`: `# To fully automate on-push, promote these files to tracked and add a GitHub Action.`). The gitignore itself acknowledges automation is missing. Preview publishing is entirely manual.

Recommended fix: Either promote the preview files to tracked (with a workflow to push on merge to `main`), or document the manual trigger explicitly in CONTRIBUTING.md. Current state is: an important pre-release step with no automation and a TODO comment in `.gitignore`.

## Low (nice-to-have)

**16. No semantic version bump automation — version string in `CHANGELOG.md` is manually maintained.**

There is no `make bump-version`, `make release`, or GitHub Action that increments the semver in CHANGELOG.md headers and creates a git tag. Version strings (`0.7.1-alpha`) are manually duplicated across CHANGELOG.md, CONTRIBUTING.md (milestone table), MEMORY.md, and checkpoint.md.

Recommended fix: Add a `make tag-release VERSION=x.y.z` target that: (a) updates the `[Unreleased]` section header in CHANGELOG.md, (b) creates a signed git tag, (c) pushes. Optionally use `commitizen` or `semantic-release` for full automation.

**17. `pre-commit-hooks` rev is pinned to `v4.5.0` — a newer version (`v4.6.0`) is available.**

`.pre-commit-config.yaml` pins `pre-commit-hooks` at `v4.5.0` and `gitleaks` at `v8.18.4`. These are not outdated enough to pose security risk, but running `pre-commit autoupdate` before v0.8.0 is good hygiene.

**18. No `make guardian-test` target in the main repo — Guardian tests are entirely in the subpackage.**

`packages/guardian/` has its own test suite, but there is no Makefile target that runs it from the project root. `make test-guardian-live` (mentioned in MEMORY.md) runs Docker lifecycle tests, but the Guardian unit tests (`packages/guardian/tests/`) are not included in `make test`, `make ci`, or `make test-all`. A change to `src/security/guardian.py` (the embedded guardian) could break the sidecar behavior without any local test catching it.

**19. `sync-guardian.yml` uses `--force` push to the public repo without a prior integrity check.**

`sync-guardian.yml` line 69: `git push guardian-public HEAD:main --force`. Every guardian sync force-pushes the public repo. If the subtree split or filter-repo produces an unexpected result (e.g., an empty commit, a regression, an accidental secret in a commit message), it overwrites the public history without a pre-push review step.

This is acceptable for an automated sync but should be noted: the `GUARDIAN_REPO_PAT` secret in `jp-cruz/LegionForge` will need to be re-created in `LegionForge/LegionForge` after the v0.8.0 repo transfer, or the sync will silently fail.

## Proposed GitHub Actions Workflows

**1. `ci.yml` — Full CI gate on PRs to `main` and `dev`**
- Trigger: `pull_request: branches: [main, dev]`
- Steps: checkout → Python 3.11 → `pip install -r requirements.lock` → Black check → smoke tests → (on `main` PRs only) testlab suite → bandit → pip-audit → URI secret scan
- Rationale: Replaces/supplements smoke-only gate. Locks to requirements.lock for reproducibility. Catches testlab/UI regressions before `main`.

**2. `docker-build.yml` — Validate Docker image builds on PRs to `main`**
- Trigger: `pull_request: branches: [main]`, path filter: `Dockerfile*`, `requirements.txt`, `src/**`, `config/**`
- Steps: checkout → docker build (gateway + guardian, no push) → smoke test inside gateway container
- Rationale: Catches broken Dockerfiles and missing requirements before release.

**3. `lint.yml` — Black format enforcement on all PRs**
- Trigger: `pull_request: branches: [main, dev]`
- Steps: checkout → Python 3.11 → `pip install black>=26.3.1` → `black --check src/ tests/ config/`
- Rationale: Enforces what pre-commit only catches locally. Single fast job (~10s).

**4. `dep-audit.yml` — Automated dependency CVE scan (weekly + on requirements.txt change)**
- Trigger: `schedule: cron: '0 9 * * 1'` + `push: paths: [requirements.txt]`
- Steps: checkout → Python 3.11 → `pip install pip-audit` → `pip-audit -r requirements.txt`
- Rationale: Weekly scheduled scan catches newly-published CVEs without a code change triggering it.

**5. `release.yml` — Tag + changelog on manual dispatch**
- Trigger: `workflow_dispatch: inputs: version`
- Steps: verify `[Unreleased]` section non-empty → update CHANGELOG.md header → create signed tag → push tag
- Rationale: Eliminates the manual version-bump steps that are currently spread across multiple files.

**6. `preview-sync.yml` — Automated preview push on merge to `main`** (only after docs/public/ is unignored)
- Trigger: `push: branches: [main]`, `paths: [docs/public/**]`
- Steps: checkout → push `docs/public/` to `jp-cruz/LegionForge` via PAT
- Rationale: Replaces `make publish-preview` manual step. Requires unignoring `docs/public/` first.

## Already Good

- **Pre-commit hook suite is comprehensive** — gitleaks (secret scanning), trailing whitespace, YAML/JSON validation, large file check, merge conflict detection, no-commit-to-main guard, Black formatting, and custom JS syntax check. This is a well-designed local gate.
- **`make ci` is a genuine full gate** — the three-suite ordering (smoke → testlab → UI) correctly catches the cross-suite asyncio event loop pollution footgun that is documented in CLAUDE.md. The gate includes static analysis (bandit), dependency CVE scanning (pip-audit), and a URI secret pattern check. Most projects skip two or three of these.
- **Guardian Docker has a proper `HEALTHCHECK`** — `guardian/Dockerfile` includes a `HEALTHCHECK` directive, as does the `docker-compose.yml` guardian service. This is correct and production-ready.
- **`requirements.lock` exists and is pip-compile generated** — the tooling for reproducible builds is in place (`make lock`, `make install-locked`). Only CI needs to be updated to use it.
- **Secret management is correctly kept out of CI** — no hardcoded secrets in workflow files. The sync workflow uses `secrets.GUARDIAN_REPO_PAT` correctly with a PAT length check before use.
- **`review-prep` is a thorough 7-step gate** — Black check, full test suite, bandit, CVE scan, secret pattern scan, new-dependency detection, and file scope check. This is more comprehensive than most project PR checklists.
- **Bandit nosec suppressions are all documented** — every `# nosec` in the source includes the specific rule ID (e.g., `B108`, `B104`, `B608`) and a comment explaining the suppression. This is correct practice.
- **`.gitignore` correctly excludes all secret patterns** — `.env.*` (with `!.env.example` carve-out), `*.key`, `*.pem`, `secrets/`, `.claude/`, `CLAUDE.md`, personal notes. No obvious gaps.
- **`sync-guardian.yml` author rewrite is correct** — the `--email-callback` approach in `git filter-repo` correctly remaps the private account identity before every public push, ensuring jp-cruz never appears in the public Guardian repo.
